什么是单例模式

单例模式，属于创建型设计模式。即一个类只有一个实例，一次实例化之后对整个系统提供这个实例对象。单例模式可以减少创建对象的内存开支，避免对资源的多重占用(比如Hibernate中的sessionFactory)

单例模式的要素

  • 构造方法私有化，即外界无法使用new关键字来创建对象
  • 通过new关键字在本类中创建一个对象
  • 定义一个公有的方法，外界可以通过这个方法获取该类的一个对象，注意因为我们需要通过这个方法来获取对象，所以这个公共方法应该是Static修饰的，可以通过类名.方法名来调用

饿汉式

public class Singleton {
    //因为构造方法是静态的，所以这个属性也必须是静态的
    //并且被static修饰后这个属性会在类加载的时候就被初始化，否则非静态成员变量只有创建对象的时候才会被初始化
    private static Singleton singleton = new Singleton();
    private Singleton() {
    
    }
    public static Singleton getInstance() {
         return singleton;
    }
}

懒汉式

public class Singleton {
    
    private static Singleton singleton;
    private Singleton() {
    
    }
    public static Singleton getInstance() {
        //懒汉式是延迟加载，即需要的时候才创建对象
        if(singleton==null){
            singleton= new Singleton();
         }
         return singleton;
    }
}

上面懒汉式的单例实现了延迟加载，但是并不能保证只有一个实例，因为存在线程安全问题。假如线程A执行到了new Singleton();此时还没有创建对象，这时候线程B会判断singleton==null是true，于是也会进入分支去创建对象，这样内存中就存在了两个实例了。我们可以通过synchronized关键字来实现线程安全。

public class Singleton {
    private static Singleton singleton;
    private Singleton() {

    }

    public static Singleton getInstance() {
        //第一个空判断来判断是否需要加锁，如果对象已经被创建了，就不会对代码进行加锁，避免了不必要的加锁对性能的影响
        if (singleton == null) {
             //多线程模式下，会有多个线程同时进入这个分支,首个获取锁的线程创建对象之后，其他线程会获取锁，继续创建对象，所以需要第二次空判断
             synchronized (Singleton.class) {
                 if (singleton == null) {
                      singleton = new Singleton();
                 }
             }
         }
         return singleton;
    }
}

上述代码有一个BUG，可能会返回一个初始化不完全的对象

Singleton singleton = new Singleton();

首先，我们要明白创建对象在JVM中并不是一个原子操作，主要由以下三步组成

  1. 在堆中分配了一部分空间
  2. 调用Singleton的构造方法初始初始化对象
  3. 把singleton对象指向堆中分配好的空间，只要完成了这一步，那么这个对象就是非空的

编译器会对上述流程进行优化重排，也就是说上面三步的执行顺序是随机的。如果说首个获取线程锁的线程正在创建对象，并且创建对象的顺序是132，在执行到3的时候另有一个线程要获取对象，这时候判断singleton已经不是空了，就会返回这个对象，这时候它就拿到了一个初始化不完全的对象。所以我们需要在属性上添加关键字volatitle，这个关键字可以保证创建对象的指令顺序不会重排。下面这就是完整的，我们经常说的双重校验锁(DOUBLE CHECK LOCKING)

双重校验

public class Singleton {
    private static volatile Singleton singleton;
    private Singleton() {
    }
    public static Singleton getInstance() {
         if (singleton == null) {
             synchronized (Singleton.class) {
                 if (singleton == null) {
                      singleton = new Singleton();
                 }
             }
         }
         return singleton;
    }
}

看到这里，有的老哥可能会有疑问，既然饿汉式的又简单又没有线程安全问题，那么我们为什么要这么麻烦呢，那么如果我们创建单例对象的时候会根据不同情况来做一些初始化的操作(给一些属性赋值)，这时候饿汉的单例模式就不适用了。而懒汉式则可以在getInstance方法中添加参数来给对象赋初始值。并且，懒加载可以一定程度上减轻系统的负担。

下面再介绍一种相当优雅的单例

静态内部类单例

public class Singleton {

    private static class CreateInstance {
         private static Singleton singleton = new Singleton();
    }

    private Singleton() {

    }

    public static Singleton getInstance() {

         return CreateInstance.singleton;
    }
}

静态内部类也同样的实现了懒加载，只有当我们调用getInstance方法的时候才会去加载内部类并初始化Singleton对象。并且JVM的类加载机制可以保证类在加载的过程中是线程互斥的，所以也不会有线程安全问题。但这个方式利用了JVM的机制，所以不能保证其他语言也是通用的。

还有一种简单粗暴的方法,并且也是线程安全的，大佬说这样好：功能完整、使用简洁、无偿地提供了序列化机制、在面对复杂的序列化或者反射攻击时仍然可以绝对防止多次实例化等优点，单元素的枚举类型是实现Singleton的最佳方法

枚举单例

public enum SingletonEnum {
    //枚举里的属性相当于Singleton的实例,注意后面有一个分号
    //这个属性默认是 Final Static的，所以可以保证是线程安全的，反序列化俺现在也没搞懂
    INSTANCE;
    private Singleton singleton;

    //枚举的构造方法是默认私有的
    SingletonEnum() {
         singleton = new Singleton();
    }
    public Singleton getInstance() {
         return singleton;
    }
}

通过下面的方法来获取Singleton的实例

Singleton singleton =SingletonEnum.INSTANCE.getInstance();
