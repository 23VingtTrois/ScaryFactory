AOP

什么是面向切面编程

在软件开发中，散布于应用多处的功能被称为横切关注点，比如说日志、安全和事务管理。横切关注点从概念上是与应用的业务逻辑分离的，但往往会直接嵌入到应用的业务逻辑中去。有的项目可能会通过一个大而全的父类，让所有的代码继承这个父类，从而避免了重复的代码散落在各处，但这样会造成继承的混乱，整个应用都使用相同的基类，会导致一个脆弱笨重的对象体系，不利于后期的升级和维护。把这些横切关注点与业务逻辑相分离正是面向切面编程(AOP)要解决的问题，在使用面向切面编程的时候，我们依然会在一个地方来定义通用的功能，但是可以通过声明的方式定义这个功能要在何时以何种方式在何处应用，而无需修改受影响的类。横切关注点可以被模块化为特殊的类，这些类被称为切面(aspect)。这样做可以使业务模块更加简洁，只需要关注自身的业务，而共通的功能则被转移到切面中了，将单独的业务逻辑和共通功能逻辑解耦。

定义AOP术语

通知（Advice）

切面的目标被称为通知，或者说通知是切面要完成的工作，再直白一点，通知里面包含了我们要实现的具体逻辑还有逻辑被调用的时间，这个被调用的时间指的是，某个方法被调用之前？之后？或者是之前之后都调用，还是只在方法抛出异常的时候调用？Spring切面可以应用5种类型的通知：

  • 前置通知（Before）
    在目标方法被调用之前调用通知功能
  • 后置通知（After）
    在目标方法完成之后调用的通知，无论方法时候发生异常，都会在方法后面调用，此时不会关心方法的输出，也获取不到方法的返回值
  • 返回通知（After-returning）
    在目标方法成功执行之后调用的通知，可以访问到方法的返回值
  • 异常通知（After-throwing）
    在目标方法抛出异常后调用通知
  • 环绕通知（Around）
    通知包裹了被通知的方法，在被通知的方法调用之前和调用之后执行自定义的行为

如果一个方法只被一个Aspect类拦截，正常执行的通知执行顺序是
环绕通知->前置通知->目标方法->环绕通知->后置通知->返回通知
发生异常的时候
环绕通知->前置通知->目标方法->后置通知->异常通知

连接点（Join point）

应用中所有可以用来应用通知的“时机”我们将其称为连接点，连接点是在应用执行过程中能够插入切面的一个点，这个点可以是调用方法时、抛出异常时，甚至是修改一个字段时。切面代码可以利用这些点插入到应用的正常流程中去，并添加新的行为。

切点（Pointcut）

一个切面并不需要通知应用的所有连接点。切点是用来缩小切面所通知的连接点的范围，如果说通知定义了切面“做什么”和“什么时候”的话，那么切点就定义了“在哪”，切点的定义会匹配通知所要织入的一个或多个连接点。我们通常使用明确的类和方法名称，或是利用正则表达式定义所匹配的类和方法名来指定切点。

切面（Aspect）

切面是通知和切点的结合。通知和切点共同定义了切面的全部内容：它有什么功能，在何时何处完成其功能

引入（Introduction）

引入允许我们向现有的类添加新的方法或属性。我自己理解这个概念就是相当于动态代理对被代理对象的功能增强

织入（Weaving）

织入是把切面应用到目标对象并创建新的代理对象的过程。切面在指定的连接点被织入到目标对象中。在目标对象的生命周期里有多个点可以进行织入：

  • 编译期
    切面在目标类编译时被织入，这种方式需要特殊的编译器。AspectJ的织入编译器就是以这种方式织入切面的
  • 类加载期
    切面在目标类加载到JVM时被织入，这种方式需要特殊的类加载器，它可以在目标类被引入应用之前增强该目标类的字节码。AspectJ 5 的加载时织入就支持以这种方式织入切面
  • 运行期
    切面在应用运行时的某个时刻被织入。一般情况下，在织入切面时，AOP容器会为目标对象动态的创建一个代理对象。Spring AOP就是以这种方式织入切面的

通知包含了需要用于多个应用的横切行为；连接点是程序执行过程中所有能够应用通知的点；切点定义了通知被应用的具体位置（哪些连接点）；切面是通知和切点的结合；织入是切面能完成功能的手段（生成代理对象的过程）；引入是面向切面编程的功能之一（动态代理功能增强）

Spring对AOP的支持

并不是所有的AOP框架都是相同的，它们在连接点模型上会有强弱之分，有些允许在字段修饰符级别应用通知，而另一些只支持与方法调用相关的连接点。它们的织入切面的方式与时机也不同，我们现在只关注Spring AOP，Spring提供了四种类型的AOP支持：

  • 基于代理的经典Spring AOP
    前三种都是Spring AOP实现的变体，Spring AOP构建在动态代理的基础上，因此Spring对AOP的支持局限于方法拦截，经典Spring AOP过于笨重和复杂，用的不多
  • 纯POJO切面
    借助Spring的aop命名空间，我们可以将纯POJO转换为切面。实际上这些POJP支持提供了满足切点条件时所需要调用的方法，但这种技术需要XML配置
  • @AspectJ注解驱动切面
    Spring借鉴了AspectJ的切面，以提供注解驱动的AOP。本质上它依然是Spring基于代理的AOP，但是编程模型几乎和AspectJ完全一致，使用这种方式可以不使用XML来完成功能
  • 注入式AspectJ切面
    如果你的AOP需求超过了简单方法调用（比如构造器或属性拦截），那么你需要考虑使用AspectJ来实现切面
Spring通知是Java编写的

Spring所创建的通知都是用标准的Java类来编写的。并且，定义通知所应用的切点通常会使用注解或在Spring配置文件里采用XML来编写。而AspectJ虽然现在支持基于注解的切面，但是AspectJ最初是以Java语言扩展的方式实现的，通过特有的AOP语言可以获得更强大和细粒度的控制，以及丰富的AOP工具集

Spring在运行时通知对象

通过在代理类中包裹切面，Spring在运行期间把切面织入到Spring管理的bean中。代理类封装了目标类，并拦截被通知方法的调用，再把调用转发给真正的目标bean，当代理拦截到方法调用时，在调用目标bean方法之前，会执行切面逻辑。直到应用需要被代理bean时，Spring才会创建代理对象，比如使用ApplicationContext的时候，在ApplicationContext从BeanFactory中加载所有bean的时候，Spring才会创建被代理的对象

Spring只支持方法级别的连接点

Spring的AOP是基于动态代理，所以就只支持方法级别的连接点，但并不是所有的AOP框架都是这样的，其他AOP框架还可以支持比如字段和构造器级别的连接点

通过切点来选择连接点

前面我们说过，切点描述了切面将会插入到哪些连接点。在Spring中，我们使用AspectJ的切点表达式语言来定义切点，Spring仅仅支持AspectJ切点指示器的一个子集

AspectJ指示器描述
args()限制连接点匹配参数为指定类型的执行方法(通过参数重载来区分各种方法)
@args()限制连接点匹配参数由指定注解标注的执行方法
execution()用于匹配是连接点的执行方法
this()限制连接点匹配AOP代理的bean引用为指定类型的类
target()限制连接点匹配目标对象为指定类型的类
@target()限制连接点匹配特定的执行对象，这些对象对应的类要具有指定类型的注解
within()限制连接点匹配指定的类型
@within()限制连接点匹配指定注解所标注的类型
@annotation限定匹配带有指定注解的连接点

观察上面的表格我们可以发现，除了execution之外，其他指示器的描述都是限制。实际上Spring中确实只有execution是用来实际执行匹配的，这也是我们编写切点时最重要的指示器，在它的基础上，我们使用其他指示器来限制所匹配的切点。

package SpringAspect;

public interface Performance {
    public void perform();
}
编写切点

我们想编写Performance的perform()方法触发通知

execution(* SpringAspect.Performance.Performance.perform(..)) 这是一个其诶单表达式，第一个星号表明返回任意类型，后面紧跟方法所属的类和方法名，方法后面表示使用任意参数
使用execution()指示器选择Performance的perform()方法，方法表达式以“*”号开始，表明我们不关心方法的返回值类型，然后是全限定名和方法名，方法的参数列表我们使用了两个点号(..)来表明其诶单选择任意的perform方法，无论该方法的入参是什么

在切点中选择bean

Spring还有一个bean()指示器，它允许我们在切点表达式中使用bean的ID来标识bean。bean() 使用bean ID或bean名称作为参数来限制切点只匹配特定的bean
execution(* concert.Performance.perform()) and bean ('woodstock')
执行Performance的perform()方法时应用通知，但是限定bean的ID为woodstock

使用注解来创建切面

定义切面
package SpringAspect;

import org.aspectj.lang.ProceedingJoinPoint;
import org.aspectj.lang.annotation.*;

@Aspect
public class Audience {

    @Pointcut("execution(* SpringAspect.Performance.perform())")
    public void performance() {

    }

    @Around("performance()")
    public void watchPerformance(ProceedingJoinPoint obj) {
        System.out.println("~~~~~~~~~~AROUTNBEFORE");
        try {
            Object proceed = obj.proceed();
        } catch (Throwable throwable) {
            throwable.printStackTrace();
        }
        System.out.println("~~~~~~~~~~AROUTNAFTETR");
    }

    @Before("performance()")
    public void silenceCellPhones() {
        System.out.println("Silencing cell phones");
    }

    @Before("performance()")
    public void taskSeats() {
        System.out.println("Taking seats");
    }


    @AfterReturning("performance()")
    public void applause() {
        System.out.println("CLAP CLAP !!!");
    }

    @AfterThrowing("performance()")
    public void demandRefund() {
        System.out.println("Demanding a refund");
    }
}

Audience类使用@Aspect注解进行了标注。这个注解表明了这个类不但是一个POJO，还是一个切面。Audience类中的方法都使用注解来定义切面的具体行为。使用@Pointcut注解可以在切面内定义可重用的切点。performance方法的实际内容并不重要，事实上应该是空的，他只是一个表示，用来给@Pointcut注解依附。@Around标注的环绕通知和其他有些不同，它需要接收一个ProceedingJoinPoint类作为参数，并且在方法中调用**proceed()**方法，再通知中通过这个方法来调用被通知方法，如果不调用这个方法，被通知方法可以一直被阻塞。

package SpringAspect;

import org.springframework.context.annotation.Bean;
import org.springframework.context.annotation.ComponentScan;
import org.springframework.context.annotation.Configuration;
import org.springframework.context.annotation.EnableAspectJAutoProxy;

@Configuration
@EnableAspectJAutoProxy
@ComponentScan(basePackageClasses = Performance.class)
public class ConcertConfig {
    @Bean
    public Audience audience() {
        return new Audience();
    }
}

要激活这个切面还需要在JavaConfig里面进行配置，使用 EnableAspectJAutoProxy来开启AspectJ自动代理，并把Audience声明为bean，AspectJ自动代理会为标记了@Aspect注解的bean创建一个代理对象，这个代理对象会围绕所有该切面的切点所匹配的bean

处理通知中的参数

首先我们要明白，通知方法是否有参取决于被通知方法，如果被通知的方法含有参数，那么切面也需要含有参数，我们可以在切点表达式总声明参数，并将这个参数传递到通知中

package SpringAspect;

import org.aspectj.lang.ProceedingJoinPoint;
import org.aspectj.lang.annotation.*;
import org.springframework.stereotype.Component;

@Component
public class Audience {

    @Pointcut("execution(* SpringAspect.Performance.perform(String))" + "args(parameter)")
    public void performance(String parameter) {

    }

    @Before("performance(parameter)")
    public void silenceCellPhones(String parameter) {
        System.out.println("Silencing cell phones");
    }
}

我们需要关注的是切点表达式中的args(parameter)限定符，它表明传给perform()方法的String类型的参数也会传递到通知中去，并且，参数的名称parameter也与切点方法签名中的参数相匹配。这个参数会传递到通知方法中，这个通知方法是通过@Before注解和命名切点performance(parameter)来定义的，切点定义中的参数于切点方法中的参数名称是一样的，这样就完成了从命名切点到通知方法的参数转移。

通过注解引入新的功能

首先我们要明白Java是静态语言，一旦类编译完成了，我们就很难为类继续添加新的功能。但通过上面切面的例子，我们虽然没有为对象增加任何新的方法，但是我们已经为对象拥有的方法增加了新的功能。切面可以为为现有的方法增加额外的功能，当然也可以为对象增加新的方法

当引入进口的方法被调用的时候，代理会把此调用委托给实现了新接口的其他对象，实际上，一个bean的实现被拆分到了多个类中

package SpringAspect;

import org.aspectj.lang.annotation.Aspect;
import org.aspectj.lang.annotation.DeclareParents;
import org.springframework.stereotype.Component;

@Component
@Aspect
/*
首先，给类添加方法需要用到切面，所以这个是一个切面，并需要声明它，把它交给IOC容器
*/
public class AudienceIntroducer {
    @DeclareParents(value = "SpringAspect.Performance+", defaultImpl = EncoreableImp.class)
    public static Encoreable encoreable;
}
//========================================================================
package SpringAspect;

import org.springframework.stereotype.Component;
/*
被引入接口的实现类
*/
@Component
public class EncoreableImp implements Encoreable {
    @Override
    public void performEncore() {
        System.out.println("我是添加的方法");
    }
}
//========================================================================
package SpringAspect;

import org.springframework.context.ApplicationContext;
import org.springframework.context.annotation.AnnotationConfigApplicationContext;

public class AspectJDemo {

    public static void main(String[] args) {

        ApplicationContext context = new AnnotationConfigApplicationContext(ConcertConfig.class);
        //首先我们获取的依然是Performance对象，这个对象里只实现了perform()方法
        Performance performanceImp = (Performance) context.getBean("PerformanceImp");

        performanceImp.perform();
        //但是我们可以直接把Performance强转型成一个Encoreable类型，并调用它的方法，完成了对Performance类的方法添加
        Encoreable encoreable = (Encoreable) performanceImp;
        encoreable.performEncore();
    }
}

@DeclareParents注解由三部分组成：

  • vaule
    这个属性指定了哪种类型的bean要引入接口(或者说是哪种类型的bean需要增加功能)，在上面的例子也就是所有实现Performance的类型（注意后面有个加号，表示是Performance的所有子类型，而不是Performance类本身，这个加号是个大坑，一开始漏了，想了半天）
  • defaltImpl
    这个属性注定了为引入功能提供实现的类
  • @DeclareParents
    注解所标注的静态属性指明了要引入的接口，在这里我们要引入的是Encoreable接口（这里我还有一点地方没想明白，为什么这个属性必须是静态的）

首先，我们以动态代理的角度来理解这个方法增强，切面相当于是代理对象，而切点相当于是被代理对象，我们获取的其实都是代理对象，代理对象有着被代理对象的所有方法，同时，我们也可以在代理对象中添加一些别的方法，这样就变相的完成了为被代理对象增加新的方法，那么怎么往代理对象中添加方法呢，应该是代理对象同时也代理了添加方法的那个类。而我们观察这个注解本身，有一个parents的意思，可以理解成会给一个类增加一个新的父类，而这个类，和作为功能被引入的类都属于这个父类，自然就可以由同一个代理对象代理。

上面我们介绍了基于注解的方式来声明切面，但这样做有一个非常明显的劣势：你必须能够为你的类来添加注解，换句话来说，你必须有源码，如果没有源码的话，我们只能使用XML配置文件来声明切面

在XML中声明切面

    <aop:config>
        <aop:aspect id="audience">
            <!--定义切点-->
            <aop:pointcut id="performance" expression="execution(* SpringAspect.Performance.perform())"/>
            <!--环绕通知-->
            <aop:around
                    pointcut-ref="performance"
                    method="watchPerformance"/>
            <!--前置通知-->
            <aop:before
                    pointcut-ref="performance"
                    method="silenceCellPhones"/>
            <!--后置通知-->
            <aop:after
                    pointcut-ref="performance"
                    method="taskSeats"/>
            <!--返回通知-->
            <aop:after-returning
                    pointcut-ref="performance"
                    method="applause"/>
            <!--异常通知-->
            <aop:after-throwing
                    pointcut-ref="performance"
                    method="demandRefund"/>
        </aop:aspect>

        <aop:aspect>
            <!--为类添加新的功能-->
            <!--types-matching 用于指定匹配哪些接口的所有子类型-->
            <!--implement-interface 要引入的接口-->
            <!--default-impl 引入接口的默认实现，这里使用全路径名-->
            <!--delegate-ref 引入接口的默认实现，这里使用引用-->
            <aop:declare-parents
                    types-matching="SpringAspect.Performance+"
                    implement-interface="SpringAspect.Encoreable"
                    default-impl="SpringAspect.EncoreableImp"
                    delegate-ref="encoreableImp"
            />
