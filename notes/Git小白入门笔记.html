Git小白入门笔记

Git是目前世界上最先进最牛逼的分布式版本控制系统（没有之一）。相对比于集中式版本控制，比如SVN，分布式系统没有中央服务器，每台电脑都保留了所有的历史版本，相对于集中式更为安全，在没有网络的情况下也可以进行版本的回退

Git的结构

  • 本地库：历史版本
  • 暂存区：未提交代码
  • 工作区：写代码的地方，但这个工作区一定要是Git的目录下的

我们把文件往Git版本库里添加的时候，是分两步执行的：第一步是用git add把文件添加进去，把文件修改添加到暂存区；第二步是用git commit提交更改，把暂存区的所有内容提交到当前分支。我们创建Git版本库时，Git自动为我们创建了唯一一个master分支，git commit就是往master分支上提交更改。需要提交的文件修改通通放到暂存区，然后一次性提交暂存区的所有修改。这个暂存区是为啥呢？咱们用SVN提交代码的时候，可以在很多修改的文件里面打钩，然后一并提交，但是Git用命令行它不方便啊，一个一个的提交太不方便了，一起提交又容易出错，所以就弄了一个暂存区。咱们把要提交的文件先全部放在暂存区，这样还可以再检查一遍是否有遗漏或者多提交

设置签名

相当于SVN的账户，这样大家可以通过看签名知道是谁进行了操作，项目级别优先系统级别，二者必须设置一个

  • 项目级别：仅某个项目内使用
    git config user.name zhj
    git config user.name zhj@gmail.com
  • 系统用户级别：登陆当前操作系统的用户范围
    git config--global user.name zhj
    git config--global user.name zhj@gmail.com

创建版本库和提交文件

  • git init
    在一个适合的目录内，执行上面的命令，这个目录就会被初始化为Git的版本库。版本库又名仓库，英文名repository，这个目录里面的所有文件都可以被Git管理起来，每个文件的修改、删除，Git都能跟踪，以便任何时刻都可以追踪历史，或者在将来某个时刻可以还原，初始化好的目录下有一个.git的隐藏文件，这个目录是Git来跟踪管理版本库的

  • git add test.txt
    把一个名为test.txt的文件由工作区提交到暂存区，这个命令有两个常用的参数，一个是-A：提交所有变化，包括了添加、修改、删除，另一个是-u：这个指令会把所有修改提交，但是对新增文件不作处理，这里所说的提交都是提交到暂存区，使用上面两个参数的时候就不用指定文件了

  • git commit -m"message" test.txt
    把一个名为test.txt的文件由工作区提交到暂存区，-m后面的是提交的版本信息，相当于SVN里面的log，可以后面加上文件名来只提交某一个文件，也可以不加，不加的意思是把暂存区里所有的文件提交到版本库。这时候如果一个文件修改了，但是没有add，那么就不会提交。commit有两个常用参数，一个是-o：only 只提交某一特定的修改，你后面加上了文件名默认就是这个。-a：提交所有变化，你只写commit 默认就是这个

  • git status
    我们使用这个命令用来查看，仓库当前的状态，可以看我们暂存区有哪些文件没提交，有哪些文件我们修改了但没有提交到暂存区等

查看版本信息

  • git log --pretty=oneline
    查看所有历史记录，后面加上--pretty=oneine用来提高可读性，也可以这样写：git log --oneline，会把版本号只显示前七位，Git的版本号不同于SVN，采用了hash，这个指令还可以在最后面加上文件名用来只查看某个文件的所有信息
  • git log --stat
    查看最近的提交修改，这个指令可以在最后面加上文件名来只查看某个文件的所有信息
  • git show <hashcode> <filename>
    查看某个文件在某次提交中的修改，这指令，也可以查看某一次修改了哪些文件或者这个文件经历了哪那些修改，只要去掉一个参数就行
  • git reflog
    git log只能显示当前版本和它比它更老的版本，而git reflog可以显示所有版本，包括比它新的版本（当前版本有可能是由更新的版本退回来的）

版本比较

  • git diff file.txt
    对比工作区和暂存区的file文件
  • git diff HEAD file.txt
    对比工作区和版本库里HEAD分支的file文件
  • git diff --staged file.txt
    对比暂存区和版本库里HEAD分支的file文件

如果不加文件名，则是比较所有文件

版本回退

git reset通过操作HEAD指针来实现历史记录的前进和回退，回退的时候要小心工作区是否有未提交代码

  • git reset --hard e2a45b2
    基于版本号，可以把HEAD的指针移动到任意版本
  • git reset --hard HEAD^
    使用^只能回退版本，一个^退一步，n个^表示退n个版本，当然没人写n个^
  • git reset --hard HEAD~3
    使用~只能回退版本，波浪线后面跟的表示后退几个版本，表示后退3个版本

reset有三个参数

  • --soft
    仅仅在本地库移动HEAD指针
  • --mixed
    在本地库移动HEAD指针，重置暂存区，重置的意思是把暂存区变成和本地库的版本一样，会清除多余文件并覆盖修改
  • --hard
    在本地库移动HEAD指针，重置暂存区和工作区，这个地方有点危险！如果你工作区有一些代码是有用的且没提交，直接就没了！

撤销修改

有时候我们会发现我们把代码给改乱了，SVN我们会使用revert来把本地的代码还原到和版本库一样，Git也有类似的操作，不过Git因为有缓存区，所以情况稍微复杂一点点

  • git checkout -- file.txt
  • git restore file.txt 推荐使用
    工作区回退，把在工作区的修改全部撤销。分为两种情况，一个是这个文件自修改后还没有被放到暂存区，那么这个指令可以把文件恢复到和版本库一模一样的状态。另一种情况是这个文件已经添加到暂存区，之后又做了修改，那么这个指令可以把工作区的文件变成添加到暂存区之后的状态。总结来说就是让工作区文件回到最近的一次git add或git commit时的状态

还有一种情况是，我们已经把代码提交到暂存区，这时候我们想要撤销，那我们就需要先撤销暂存区，再撤销工作区

  • git reset HEAD file.txt
  • git restore --staged file.txt 推荐使用
    暂存区回退

再一种情况就是已经提交到版本库了，那么我们就需要去做版本的回滚，还有一种情况是你提交到了远程版本库，这就有些麻烦了，咱们后面说

  • git rm file.txt
    删除文件，并把删除的操作提交到了暂存区，之后使用git commit就会把版本库里面对应的文件也删掉，如果只用rm file.txt那么只是删除了工作区
  • git clear
    用来删除所有没有被track的文件！所有没有被Git管理的文件都会被清理掉！这个慎用！使用前可以用clear -n来看看一眼它会删除哪些文件，慎用！论坛因为这个命令吃亏的大有人在

远程仓库

  • git remote add origin https://github.com/23VingtTrois/ScaryFactory.git
    这个指令是吧咱Git的远程库地址起一个别名，这样在用到地址的时候使用origin就可以了，使用git remote -v可以查看当前系统里有哪些别名和对应的地址
  • git push origin master
    将master分支上的内容推送到远程仓库
  • git clone origin
    从远程库克隆到本地，有点像SVN的checkout，clone的同时会自动初始化本地库
  • git pull origin 分支名
    从远程库把对应分支的代码拉取到本地，很像SVN里面的Update

分支管理

大家使用肯定有过这种情况，写了好几天的代码，但代码并没有完成，这时候上传会影响其他人工作。我们把未完成的代码放在本地，等到完全完成再上传，这样万一本地出了什么问题代码没了就GG了。分支用来解决这类问题。从主分支上拉出一条新分支，我们在新分支上开发，等待开发完成再合并到主分支。SVN也是支持分支的，但是据说创建和合并切换分支等操作慢的异常，根本没人用。关于分支的理念，有篇文章讲得很好：Git分支管理策略

  • git branch -v
    查看所有分支

  • git branch hotfix
    创建分支，分支名是hotfix

  • git checkout 分支名字
    git switch 分支名字
    切换分支，也可以使用git checkout -b 分支名，这个命令是创建并切换分支

  • git merge 分支名
    合并某分支到当前分支，我们在merge的时候可能会发生冲突，这时候我们需要手动去解决冲突并依次使用git add和git commit来提交代码，Git通常合并分支的时候会使用Fast forward模式，这种模式下删除分支之后会丢掉分支信息，我们合并分支的时候可以使用--no--ff表示禁用Fast forward，但这样合并我们需要commit，所以需要加上一个-m，我们使用git log --graph --pretty=oneline --abbrev-commit来查看分支历史

  • git branch -D 分支名
    删除分支

  • stash
    当我们本地工作还没有完成，并不能提交代码，但我们这时候需要新建一个分支来修改代码，Git在缓存区有未提交的时候偶是不能切换分支的。这时候我们可以使用git stash来把我们本地未提交的代码先保存起来。使用git stash list可以查看保存的工作现场。需要恢复的时候有两个办法：一是用git stash apply恢复，但是恢复后stash内容并不删除，你需要用git stash drop来删除；另一种方式是用git stash pop，恢复的同时把stash内容也删除了

  • git cherry-pick 版本号
    这个指令可以把某一次的提交复制到当前分支并提交，在master分支上修复的bug，当前分支肯定也有问题，使用这个命令就不需要改两遍了

标签管理

发布版本的时候，我们通常会把该版本打一个标签，这样后面我们说某一个版本就不需要说一大堆数字了，取而代之的是我们自己定义的标签，标签是一个好记的、有意义的名字，它跟某一个commit绑定在一起

  • git tag -a 标签名
    切换到需要打标签的分支上，这个标签代表的就是这个分支最近的一次commit。也可以用git tag 标签名 版本号为某个历史版本打标签，可以添加参数-m来添加说明
  • git push origin 标签名
    把本地标签推送到远程，也可以使用git push origin --tags来推送所有本地未推送的标签
  • git tag -d 标签名
    删除一个本地标签
  • git push origin : refs/tags/标签名
    删除一个远程标签
